# File Structure Conventions

This document defines the expected file and folder layout for each primary stack. When starting a project, scaffold this structure first. When adding to a project, put files where this document says they go. Consistency across projects means less cognitive overhead and faster onboarding for both humans and agents.

## React Native + Expo (Mobile)

```
app/                    # Expo Router file-based routing
  (tabs)/               # Tab navigation group
    index.tsx           # Home/default tab
    explore.tsx         # Second tab, etc.
    _layout.tsx         # Tab layout with tab bar config
  (auth)/               # Auth flow screens (not in tabs)
    login.tsx
    _layout.tsx
  _layout.tsx           # Root layout (providers, fonts, splash)
  +not-found.tsx        # 404 handler

components/             # Reusable UI components
  ui/                   # Primitive/atomic components (Button, Input, Card)
  [FeatureName]/        # Feature-specific components (MapMarker, TraceCard)

lib/                    # Business logic, utilities, clients
  supabase.ts           # Supabase client initialization
  auth.ts               # Auth helper functions
  storage.ts            # File upload/download helpers
  constants.ts          # App-wide constants (colors, spacing, config)
  types.ts              # Shared TypeScript types/interfaces

hooks/                  # Custom React hooks
  useLocation.ts
  useAuth.ts

assets/                 # Static assets (images, fonts)
```

Key conventions for React Native:
- One component per file. File name matches the default export name.
- Colocate feature-specific components near where they're used, but in a `components/` subfolder if shared across screens.
- All Supabase calls go through helper functions in `lib/`, never called directly from components.
- Types shared across more than one file go in `lib/types.ts`. Types used in only one file stay in that file.

## Next.js (Web)

```
app/                    # App Router
  page.tsx              # Home page
  layout.tsx            # Root layout (providers, metadata)
  (auth)/               # Auth route group
    login/page.tsx
    callback/page.tsx
  (dashboard)/          # Authenticated route group
    layout.tsx          # Dashboard layout (sidebar, nav)
    page.tsx            # Dashboard home
    settings/page.tsx
  api/                  # API routes (if not using Supabase Edge Functions)
    webhooks/
      stripe/route.ts

components/             # Same structure as RN above
  ui/
  [FeatureName]/

lib/                    # Same as RN above
  supabase/
    client.ts           # Browser Supabase client
    server.ts           # Server-side Supabase client
    middleware.ts        # Auth middleware helper

hooks/

public/                 # Static files served at root
```

Key conventions for Next.js:
- Separate Supabase client files for browser vs. server contexts. This is not optional — mixing them causes auth bugs.
- Route groups `(groupName)` for layout boundaries, not for URL structure.
- API routes only for webhook handlers or third-party integrations. Prefer Supabase Edge Functions or direct Supabase client calls for your own data.
- Use Server Components by default. Add `'use client'` only when the component needs interactivity (state, effects, event handlers).

## Supabase Project Structure

```
supabase/
  migrations/           # SQL migration files (generated by Supabase CLI)
    00001_initial.sql
  functions/            # Edge Functions
    process-webhook/
      index.ts
    send-notification/
      index.ts
  seed.sql              # Seed data for development
  config.toml           # Supabase local dev config
```

Key conventions for Supabase:
- Always use migrations, even for MVPs. `supabase db diff` to generate them from the dashboard, then commit.
- One Edge Function per folder. Keep them small — a function should do one thing.
- RLS policies are part of migrations, not applied ad-hoc via the dashboard.

## Agent/Automation Projects

```
agents/                 # Agent definitions
  [agent-name]/
    index.ts            # Agent entry point
    tools.ts            # Tool/function definitions
    prompts.ts          # System prompts and templates
    types.ts            # Input/output types

lib/                    # Shared utilities
  mcp.ts                # MCP client setup
  llm.ts                # LLM client initialization and helpers
  state.ts              # State management between agent turns

scripts/                # One-off or scheduled scripts
  daily-brief.ts
  ingest.ts

config/                 # Configuration
  agents.yaml           # Agent definitions and parameters
```

## Universal Rules

- **No `src/` directory.** It adds a folder level with zero information. The project root IS the source.
- **No barrel files** (`index.ts` that re-exports everything from a folder). They create circular dependency problems and confuse tree-shaking. Import directly from the file.
- **No `utils/` junk drawer.** If a utility is used by one feature, it lives in that feature's folder. If it's used across features, it goes in `lib/` with a descriptive filename, not `utils/misc.ts`.
- **Environment variables** live in `.env.local` (gitignored) and `.env.example` (committed, with placeholder values). Never commit real keys.
